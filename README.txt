Пояснение по коду (шпаргалка для защиты по ТЗ)

1) Что это за проект простыми словами
Это собственное мини-средство автоматизации тестирования на C#.
По сути, это упрощенный аналог xUnit/NUnit:
- свои атрибуты для разметки тестов,
- свои assert-проверки,
- свой раннер, который сам находит и запускает тесты,
- свой отчет с результатами.

2) Как работает проект (короткий сценарий)
1. Program.cs создает TestRunner.
2. TestRunner берет текущую сборку и через reflection ищет тестовые классы.
3. Для каждого класса выполняет lifecycle-методы и тесты.
4. Для каждого теста фиксирует статус: Passed / Failed / Error.
5. Формирует отчет и выводит его в консоль + файл TestResults.txt.

3) Что за что отвечает
- MiniTestFramework/Attributes.cs
  Атрибуты (маркеры): [TestClass], [Test], [TestCase], [BeforeEach], и т.д.

- MiniTestFramework/AssertEx.cs
  Методы проверок (assert): True, Equal, ThrowsAsync и др.
  Если проверка не прошла, бросается AssertionFailedException.

- MiniTestFramework/Exceptions.cs
  Собственные исключения фреймворка:
  TestDiscoveryException, AssertionFailedException, TestExecutionException.

- MiniTestFramework/SharedContext.cs
  Интерфейс ISharedContext для общего контекста тестового класса.

- MiniTestFramework/TestRunner.cs
  Ядро фреймворка: поиск тестов, запуск lifecycle, запуск sync/async тестов,
  обработка ошибок, сбор отчета.

- MiniTestFramework/Models.cs
  Модели результатов и сводного отчета (по тестам и по итогу).

- TestedProject/
  Бизнес-логика, которую тестируем.

- Tests/
  Набор тестов, который демонстрирует все возможности фреймворка.

4) Соответствие ТЗ по пунктам
1. Структура из 4 частей
- библиотека тестирования: MiniTestFramework/
- тестируемый проект: TestedProject/
- проект с тестами: Tests/
- программа запуска: Program.cs

2. Атрибуты
Реализованы и используются:
- для классов: TestClass, TestClassInfo, UseSharedContext
- для методов: Test, TestCase, TestInfo, BeforeAll, AfterAll, BeforeEach, AfterEach
Есть атрибуты как без параметров, так и с параметрами.

3. Проверки (минимум 10)
В AssertEx реализовано более 10 условий:
True, False, Equal, NotEqual, Null, NotNull, Contains, DoesNotContain,
Greater, Less, SequenceEqual, ThrowsAsync, IsType.

4. Обработка ошибок
Есть собственные типы исключений.
В отчете четко разделяются:
- Failed: упал assert (проверка не пройдена)
- Error: ошибка выполнения/инфраструктуры

5. Контекст тестов
Поддержаны методы подготовки/очистки:
BeforeAll/AfterAll и BeforeEach/AfterEach.

6. Асинхронность
Поддержаны тестовые методы с Task и ValueTask,
есть async-тесты и ThrowsAsync.

7. Выполнение тестов
Тесты запускаются раннером, результаты выводятся в консоль и в файл.

8. Дополнительно: Shared Context
Реализован ISharedContext + UseSharedContext,
контекст создается на класс, инициализируется и внедряется в тест.

5) Новая реальная бизнес-модель
Добавлена модель интернет-магазина: OrderPricingService.
Она считает итог заказа с учетом:
- суммы позиций,
- скидки,
- порога бесплатной доставки,
- стандартной доставки,
- приоритетной наценки,
- округления денежных значений.

Что проверяют тесты OrderPricingService:
- обычный заказ,
- бесплатная доставка при достаточной сумме,
- влияние скидки на порог бесплатной доставки,
- приоритетная доставка,
- валидация некорректных данных,
- граничные значения скидки,
- округление денег.

6) Важные формулировки для устных вопросов
- Почему reflection?
  Чтобы автоматически находить тесты по атрибутам без ручного списка.

- Чем Failed отличается от Error?
  Failed = бизнес-проверка не прошла (assert).
  Error = техническая ошибка выполнения теста или раннера.

- Зачем lifecycle?
  Отдельно настраивать окружение до/после всех тестов и до/после каждого теста.

- Зачем Shared Context?
  Хранить общие данные/ресурсы на весь тестовый класс и не пересоздавать их каждый раз.

- Как работают параметризованные тесты?
  Каждый TestCase создает отдельный запуск одного и того же метода с разными аргументами.

- Как проверяется асинхронность?
  Раннер ожидает Task/ValueTask, а в assert есть ThrowsAsync.

7) Что происходит в каждом файле
- Program.cs
  Точка входа. Создает TestRunner, запускает все тесты сборки, печатает отчет в консоль и сохраняет в TestResults.txt.

- MiniTestFramework/Attributes.cs
  Описание всех атрибутов (маркеров), по которым раннер понимает: что является тестом,
  какие методы запускать до/после тестов, где параметризованные кейсы и где shared context.

- MiniTestFramework/AssertEx.cs
  Реализация проверок. Если условие не выполнено, бросается AssertionFailedException,
  чтобы раннер пометил тест как Failed.

- MiniTestFramework/Exceptions.cs
  Иерархия исключений мини-фреймворка для корректного разделения типов ошибок.

- MiniTestFramework/SharedContext.cs
  Интерфейс общего контекста: InitializeAsync() для подготовки и Dispose() для очистки.

- MiniTestFramework/Models.cs
  Модели результата (TestCaseResult, TestRunReport), подсчет статистики,
  форматирование текста отчета для консоли/файла.

- MiniTestFramework/TestRunner.cs
  Основная логика выполнения:
  - находит тестовые классы,
  - создает экземпляры,
  - создает и внедряет shared context,
  - запускает BeforeAll/BeforeEach/Test/AfterEach/AfterAll,
  - обрабатывает исключения,
  - формирует список результатов.

- TestedProject/MathService.cs
  Простая бизнес-логика математики (сложение, вычитание, деление, async-умножение).

- TestedProject/TextService.cs
  Простая логика работы со строками и коллекциями (Join, MaybeNull, Range).

- TestedProject/OrderPricingService.cs
  Реальная бизнес-модель расчета стоимости заказа (subtotal, скидка, доставка,
  приоритетная наценка, округление, валидация входных данных).

- Tests/AppSharedContext.cs
  Конкретная реализация shared context для тестов (инициализация общих данных и очистка).

- Tests/MathServiceTests.cs
  Демонстрационный набор: обычные и async-тесты, lifecycle, TestCase,
  shared context, а также пример намеренного Failed.

- Tests/MetaAttributeTests.cs
  Проверка, что атрибуты с параметрами корректно читаются и работают через reflection.

- Tests/OrderPricingServiceTests.cs
  Набор тестов для бизнес-правил OrderPricingService (основные сценарии, границы и ошибки).

8) Короткий итог
Проект полностью демонстрирует разработку собственного средства тестирования:
атрибуты, assert, обработку ошибок, lifecycle, async, запуск и отчет.
Плюс показана реальная бизнес-модель (расчет цены заказа) с тестовым покрытием.
